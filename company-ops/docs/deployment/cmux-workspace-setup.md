## 工作区分配规则

### 工作区 1: Orchestrator（固定）

| 属性 | 值 |
|------|-----|
| 类型 | orchestration |
| 职责 | 全局协调、状态监控、人类交互 |
| 对应路径 | `company-ops/` 根目录 |
| Agent | 调度Agent |

### 工作区 1-N: 子系统工作区（动态）

根据 `subsystems/_registry.json` 中活跃子系统动态分配：

| 子系统ID | 工作区 | 状态 |
|----------|--------|------|
| 财务 | 1 | planned |
| 法务 | 2 | planned |
| products/example-product | 3 | template |

## Cloud Code 插件

项目包含一个 Cloud Code 插件 (`~/.claude/plugins/marketplaces/company-ops/`) 用于管理多工作区：

插件提供以下命令：

| 命令 | 说明 |
|------|------|
| `!Orchestrator` | 将当前工作区设为总调度 |
| `!new-subsystem <名称>` | 创建新的子系统工作区 |
| `!workspaces` | 列出所有工作区 |
| `!broadcast <消息>` | 向所有工作区广播消息 |
| `!status` | 显示系统状态 |

## 启动流程

```bash
# 1. 打开 cmux 应用
open -a cmux

# 2. 第一个工作区就是 orchestrator，运行 Cloud Code
claude

# 3. 在 orchestrator 中安装插件后，可使用以下命令创建子系统工作区
!new-subsystem 财务
!new-subsystem 法务
```

## 常用快捷键

```
{序号}-{子系统名称}
```

示例：
- `0-orchestration` - 公司总调度
- `1-财务` - 财务子系统
- `2-法务` - 法务子系统
- `3-products/example` - 示例产品

## 常用快捷键

| 快捷键 | 功能 |
|--------|------|
| `Ctrl-b c` | 创建新工作区 |
| `Ctrl-b n` | 下一个工作区 |
| `Ctrl-b p` | 上一个工作区 |
| `Ctrl-b 0-9` | 切换到指定工作区 |
| `Ctrl-b ,` | 重命名当前工作区 |
| `Ctrl-b &` | 关闭当前工作区 |
| `Ctrl-b s` | 列出所有工作区 |

## 子系统状态与工作区

| 子系统状态 | 工作区状态 | 说明 |
|------------|------------|------|
| planned | 不创建工作区 | 仅在 registry 中注册 |
| active | 创建工作区 | 正常运行 |
| inactive | 保留工作区（停止Agent） | 暂停运行 |
| error | 保留工作区 | 需要人工介入 |

## 文件与工作区映射

每个子系统工作区：
1. 工作目录：`company-ops/subsystems/{subsystem-id}/`
2. 监控文件变化：
   - `inbox/` - 接收其他子系统消息
   - `state/` - 状态更新
   - `SPEC.md` - 规范变更
3. 输出文件到：
   - `outbox/` - 发送消息
   - `state/` - 更新状态

## 监控脚本

可使用以下脚本监控系统状态：

```bash
# 检查所有子系统状态
python shared/tools/check_completeness.py --all

# 查看子系统注册表
cat subsystems/_registry.json | jq .

# 监控文件变化
fswatch -r subsystems/
```

## 注意事项

1. **资源管理**：每个工作区运行独立的 Claude Code 实例，注意内存和 API 配额
2. **状态同步**：子系统间通过文件系统（inbox/outbox）通信，非直接网络调用
3. **人工介入**：关键决策通过 `human/reviews/` 目录触发人工审核
4. **日志记录**：所有操作记录在 `.system/logs/` 目录
