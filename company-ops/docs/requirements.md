# 一人AI软件服务公司系统架构设计文档

> **说明**：本文档为系统架构设计文档，整合了需求规格与软硬件架构设计
> - 需求版本：v0.2
> - 架构版本：v0.1
> - 创建日期：2026-03-25

## 1. 项目概述

### 1.1 项目背景

本项目旨在建立一家一人股份有限公司，主营业务为软件服务（自有SaaS产品）。通过AI Agent实现公司各项工作的自动化运营，采用人机协作模式运行。

**核心愿景：**
- 公司运营全流程由不同AI Agent自主完成
- 创始人通过飞书与AI Agent协作，扮演"管理者"和"决策者"角色
- Agent分工按传统职能划分（技术研发、产品设计、市场营销、客服、财务、法务等）
- 使用文件夹系统实现部门和工作内容的层级划分

### 1.2 技术栈概览

| 组件 | 角色 | 说明 |
|------|------|------|
| **cmux** | Agent运行与调度平台 | 基于Ghostty的macOS终端，多Agent并行管理、通知聚合、CLI调度 |
| Claude Code | 主要AI Agent运行平台 | 核心CLI工具 |
| Codex | 备选AI Agent运行平台 | 辅助AI能力 |
| **cc-connect** | 本地调度客户端 | 安装在Mac上的本地Agent，负责接收飞书指令并通过CLI驱动cmux启动Agent |
| 飞书机器人 | 人机交互界面 | 任务指派、进度汇报、即时通讯 |
| OpenClaw | 通用任务处理 | 非隐私任务处理 |
| 苹果硬件设备 | 基础设施 | Mac Mini/Mac Studio |

### 3.2 可用性需求

| 指标 | 要求 | 故障恢复方案 |
|------|------|---------------|
| 系统可用性 | 99.5%（7x24运行） | 故障自动检测 + 自动重启 |

| 故障类型 | 检测方式 | 处理策略 | 恢复时间目标 |
|----------|----------|----------|--------------|

## 4. 用户故事与验收标准

### 4.1 核心用户故事

#### US-001：创建并分配任务

> 作为公司创始人，我想创建一个新任务并分配给对应的AI Agent，以便推进具体工作。

**参与者目标分析：**
- 主要参与者：公司创始人
- 目标：快速将工作意图传达给AI Agent并获得确认
- 触发：飞书消息或任务看板操作
- 最小完成条件：Agent确认接收并开始执行

**Given/When/Then 示例：**
```
Given: 用户已完成飞书认证并绑定部门
  And: 技术研发部已配置Claude Code Agent
When: 用户发送 "创建任务：开发用户登录功能，分配给技术研发部"
Then: 任务管家Bot在2秒内回复 "[T001] 任务已创建，分配给技术研发部"
And: cc-connect在5秒内启动Claude Code实例
And: Agent加载技术研发部上下文并在10秒内确认任务
And: 飞书推送 "已接收任务：开发用户登录功能，预计完成时间2小时"
```

**验收标准：**
- [ ] 可以在飞书中创建任务并选择部门和具体工作文件夹
- [ ] 任务创建后，cc-connect自动启动对应Claude Code实例
- [ ] Agent自动加载该文件夹的上下文和历史记录
- [ ] Agent在飞书中确认接收任务

#### US-002：Agent工作汇报

> 作为公司创始人，我希望AI Agent定期汇报工作进展，以便我了解任务状态。

**参与者目标分析：**
- 主要参与者：公司创始人
- 目标：实时了解任务状态，无需主动询问
- 触发：定时器（每15分钟）或关键节点事件
- 最小完成条件：收到进度汇报消息

**Given/When/Then 示例：**
```
Given: Agent正在处理任务T001 "开发用户登录功能"
When: 任务开始10分钟后
Then: Agent推送 "[T001] 进度30%，已完成：数据库设计、API定义"
And: Agent推送 "下一步计划：实现注册接口，预计30分钟"
And: 用户可以随时回复"详细"获取更多信息

Given: Agent遇到阻塞
When: Agent需要外部依赖（第三方API密钥）才能继续
Then: Agent主动@创始人 "任务T001阻塞：需要提供XX API密钥"
And: 推送消息包含"需要确认"标签
```

**验收标准：**
- [ ] Agent在任务开始、关键节点、完成任务时主动推送消息
- [ ] 任务进行中可随时在飞书询问进度
- [ ] 阻塞或需要决策时，Agent主动@创始人并说明问题
- [ ] 支持查看历史任务记录和结果

#### US-003：跨部门协作

> 作为公司创始人，当一项工作需要多个部门协作时，我希望能够协调多个Agent共同完成。

**参与者目标分析：**
- 主要参与者：公司创始人
- 目标：协调多个Agent协同完成复杂任务
- 触发：创建跨部门任务
- 最小完成条件：所有参与Agent确认分工并开始执行

**Given/When/Then 示例：**
```
Given: 用户想创建跨部门任务 "上线新产品页面"
When: 用户发送 "创建任务：上线新产品页面，需要技术和产品协作"
Then: 系统自动分解任务：产品部(原型设计) → 技术部(开发实现)
And: 产品Agent和技术Agent同时收到任务通知
And: 飞书创建群聊，包含创始人和两个Agent
And: 产品Agent先完成原型，确认后技术Agent开始开发
```

**验收标准：**
- [ ] 支持创建跨部门任务，自动分配给多个Agent
- [ ] 各Agent可共享上下文和中间产物
- [ ] 飞书群聊中多个Agent可协同对话
- [ ] 有明确的协作流程和责任分工

#### US-004：权限管理

> 作为公司创始人，我希望控制不同Agent的访问权限，保护敏感信息。

**参与者目标分析：**
- 主要参与者：公司创始人
- 目标：确保Agent只能访问授权的资源
- 触发：Agent尝试访问资源时
- 最小完成条件：未授权访问被拦截

**Given/When/Then 示例：**
```
Given: 技术Agent属于技术研发部
When: Agent尝试访问 财务部/工资表/
Then: 系统拦截访问，返回403错误
And: 创始人收到通知 "技术Agent尝试访问敏感文件夹，已拦截"
And: 日志记录：时间、Agent ID、尝试访问路径

Given: 创始人手动授权技术Agent访问设计稿
When: 用户执行 "授权技术Agent访问 产品设计部/最新设计稿"
Then: 授权成功，推送确认消息
And: 技术Agent现在可以访问该文件夹
And: 授权记录显示在权限管理面板
```

**验收标准：**
- [ ] 部门Agent默认只能访问本部门文件夹
- [ ] 需要跨部门访问时，创始人手动授权
- [ ] 敏感文件夹访问有额外验证
- [ ] 可随时查看和调整Agent权限

#### US-005：通用任务处理

> 对于不涉及隐私的通用任务，我希望使用OpenClaw处理，减轻主系统负担。

**参与者目标分析：**
- 主要参与者：系统（自动路由）
- 目标：优化资源使用，非敏感任务自动走OpenClaw
- 触发：任务创建时自动分类
- 最小完成条件：任务被正确路由到对应处理引擎

**Given/When/Then 示例：**
```
Given: 用户创建任务 "搜索竞品分析报告"
When: 任务类型被识别为 "通用研究"
Then: 系统自动路由到OpenClaw处理
And: 不占用本地Claude Code资源
And: 结果在10分钟内返回到任务面板

Given: 用户创建任务 "编写核心支付代码"
When: 任务被识别为 "敏感开发"
Then: 系统自动路由到本地Claude Code
And: 标记为高安全级别
And: 不经过OpenClaw
```

**验收标准：**
- [ ] 可配置哪些任务类型走OpenClaw
- [ ] OpenClaw处理结果自动同步
- [ ] 敏感内容不会流向OpenClaw
